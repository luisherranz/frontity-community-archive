Hi,
I’m really interested in using Frontity to build a new front, but the site is heavily customized with lots of rewrites, filters etc, so there are a few kinks I need to make sure that I can work out before I can get started on a theme.

I’ve been trying to add a custom path pattern, but I have run in to some issues and I can’t seem to find my answer in the documentation.

I used the example from the documentation and edited it to fit my purpose:

init: ({ libraries }) => {
        libraries.source.handlers.push({
          name: "country-info",
          priority: 1,
          pattern: "/:slug",
          func: async ({ route, params, state, libraries, force }) => {
            const response = await libraries.source.api.get({
              endpoint: "country-info",
              params: { slug: params.slug },
            });

            const [countryInfo] = await libraries.source.populate({
              response,
              state,
              force,
            });

            Object.assign(state.source.data[route], {
              id: countryInfo.id,
              type: countryInfo.type,
              isPostType: true,
              isCountryInfo: true,
            });
          },
        });
      },

The handler works fine when the slug matches an existing post, but when I enter a non-existing slug I get this Error:

  TypeError: Cannot read property 'id' of undefined
      at Object.func (webpack-internal:///./packages/mars-theme/src/index.js:14:558)
      at <anonymous>
      at process._tickCallback (internal/process/next_tick.js:182:7)

And the site only displays ‘Internal Server Error’.

So here’s what I’d like to know:

  1. What would you recommend that I check for in the response to see if a post was found or not?

  2. If no posts where found, what should I do to generate a proper 404 error and make the 404 page show up?

  3. Let’s say I get a result that I for some reason want to ignore on the front end, how can I make Frontity move on and try to match the next pattern in the priority chain? (As if the pattern didn’t match)

Also, thank you for making this framework. I love working with WordPress and I love working with React. I have often used React apps in websites I built in WordPress, but your framework is exactly what I have been looking for! Keep up the good work!

Fred

mmczaplinski 23 April 2020 01:31 2

Hi Fred! I’m happy that you’re enjoying working with Frontity!

As to your questions:

  1. Under the hood, source.api.get just makes a regular http request to the WP REST API using fetch, and gives you a Response back. You can check for whatever you like then, depends on what you are looking for

  2. This is a good question and I don’t have an official answer for you because we don’t have an “official” and documented way of doing that. In the framework, we are catching the errors and then checking if it’s an internal error or an error that was thrown by the framework (e.g. because one of the handlers encountered a 404).

You could follow a similar strategy and if your condition fails (e.g. post was not found) you can assign an “error object” to your current route with something like:

const response = await libraries.source.api.get({
   endpoint: "country-info",
   params: { slug: params.slug },
});

if (!response.ok) {
  const errorData = {
    isError: true,
    isReady: true,
    isFetching: false,
    is404: true,
    errorStatus: 404,
    errorStatusText: 'Page Not Found',
  };
  source.data[link] = errorData;
}

Then your UI should just show the 404 page

  1. If fetch the data from the WP REST API with libraries.source.api.get() but you don’t call libraries.source.populate() and don’t assign anything to state.source.data like mentioned in the example, the handler will not really do anything useful and move on to process the handler with the next highest priority!

Hope this helps!

luisherranz 23 April 2020 11:51 3
mmczaplinski:

the handler will not really do anything useful and move on to process the handler with the next highest priority

Not really, right now only one handler is executed per fetch. I have a proposal to change that for version 2.0 of source because it seems to be a recurring problem.

What we are doing right now is handle that logic inside the handler itself, like this: https://github.com/frontity/frontity/blob/dev/packages/wp-source/src/libraries/handlers/postType.ts#L20-L35

I hope that helps.
