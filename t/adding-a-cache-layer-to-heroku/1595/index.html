But for testing purposes, I think it’s verified that adding this addon is a good solution for adding a CDN layer in a Heroku deployment of a Frontity project

juanma 27 April 2020 08:41 10

I’m going to discard this way of adding a CDN for now, because:

  • Edge is creating a Cloudfront distribution but the way to configure Cloudfront from that plugin is kind of complex, limited and not very well documented

    • You have to use AWS CLI to configure Cloudfront from that plugin → https://devcenter.heroku.com/articles/edge#cloudfront-configuration
    • The set of actions you can do from AWS CLI to configure Cloudfront is limited to this ones → https://docs.aws.amazon.com/cli/latest/reference/cloudfront/index.html
      • it seems the update-distribution is the way to go to customize headers but it is complex:
        • https://docs.aws.amazon.com/cli/latest/reference/cloudfront/update-distribution.html
        • https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/distribution-overview-required-fields.html

  • Cloudfront itself is very powerful but very complex as well, so I think we should focus first on more easy solutions

    • https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/header-caching.html
    • https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/RequestAndResponseBehaviorCustomOrigin.html#request-custom-headers-behavior

Maybe configuring a CDN layer with a service like KeyCDN is a better option to start with
