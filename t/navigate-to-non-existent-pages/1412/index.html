🙋Get Help
chris 18 March 2020 04:28 1

We’re trying to allow users to navigate between pages within a form but are running into a few issues, the URL structure is something like:

  • /contact/
  • /contact/page2/
  • /contact/page3/

(This is an example, the form in question isn’t a contact form )

Within WP the initial /contact/ page has been created but these “sub-pages” haven’t. I’ve created a handler which will load the page data from the initial /contact/ page, accessing these pages directly works fine but when a user clicks back/forward in their browser we’re noticing within our Theme component, the router link is “undefined/” and as such returns no data required for the page e.g. Yoast, ACF, etc.

Currently when a user navigates to the next page of the form we’re using history.pushState:

let slug = "page2"; // Slug is created based off the current page the user is viewing
window.history.pushState({ data: [..] }, 'Page Title', `/contact/${slug}/`);

We’ve toyed around with both popstate and beforeunload window events to try and capture this but they seem to either get called too late (if at all) within these events we’ve tried manually setting the router:

actions.router.set("/contact/page2/");

But still have the same issue. I’m probably missing something super obvious here, but I’m not really sure what…

Ideally we’d like to just re-render the form with the new page instead of loading an entirely new page.

luisherranz 23 March 2020 08:21 7

@chris I guess what you want is all those Frontity URLs (/contact/, /contact/page2/ and /contact/page2/) pointing to the same WordPress URL: /contact, right?

That’s not something that can be done in the current version of source but we are working on a new version where that configuration will be possible.

Since we get to that point, my recommendation is to stick to a single URL and save the progress of your form in the client using localStorage, including the subpage the users are currently in. That way, if the user refreshes the page, it will be in the same place where they left off.

You can use a React hook that works similar to useState but saves the data on localStorage like this one: https://www.npmjs.com/package/use-persisted-state

Remember to include a reset button so users can go back to the subpage 1.

Another idea that just came to my mind is to bypass the Frontity’s routing for those URLs.

const Theme = ({ state }) => {
  // ...
  const isContatPage = state.router.link.startsWith("/contact/");

  return (
    <Switch>
      <ContactPage when={isContactPage} />
      <Loading when={isLoading} />
      {/* ... */}
    </Switch>
  );
}

Then, inside ContactPage, extract the page number from the URL:

const ContactPage = ({ state }) => {
  const page = /\/contact\/page(\d)/.exec(state.router.link)[1];
  // ...
};

If you do that, I think you can use actions.router.set and forth and the browser back/forward button should work.

The problem with that approach is that the SSR of the /contact/page2 URL will return a 404 because it doesn’t exist on your WordPress but you can create empty pages that match those URLs.
